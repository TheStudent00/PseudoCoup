# BUG PROFILE — kt↔py twin divergences (living document)

Purpose: one catalog of every observed divergence between the Kotlin app (ground truth) and its
transpiled Python twin. Each entry records the SYMPTOM, where it was seen, and the HYPOTHESIZED root
cause + which layer it lives in. The goal is not just a to-do list — it is to watch for CLUSTERS: when
several symptoms trace to one cause, that cause is the real target. Update status as bugs are confirmed
or fixed. Owner asked to start this so "a model forms on its own."

Layers (where a bug actually lives):
- DATA        — the starting DB / seed state differs between engines
- WIRING      — a handler/callback is dead or misrouted (code translated, behavior lost)
- PROGRESSION — workout/flow state-machine logic (advancing sets/exercises/completion)
- LAYOUT      — runtime layout/overlay/z-order (Compose-vs-kivy render wiring)
- RUNTIME     — the emulated Compose runtime silently degrading (autostub -> 0/no-op)
- VERSION     — py may predate a kt feature (WFL was statically copied at a point in time)

## NEW PRIMARY INSTRUMENT: directed scenario runner (render/scenario.py, log_152)
Stateful, scripted, ONE session — drives a real user journey (select path -> enroll program -> Home shows the
schedule -> start workout -> log sets -> finish), screenshots each step. ~17s for 18 steps vs ~93 min for the
BFS walker, and it REACHES rich app-states the reboot-per-edge walker structurally cannot (proven: Home showed
the enrolled program's weekly schedule). This is now the tool for first-few-taps debugging; the BFS walker is
for breadth coverage only.

## Emerging clusters (the "model")

- **DATA — DISSOLVED (log_150):** B01 disproven (seed correct/complete), B03 not-a-bug (RPE is correct by
  design). The DB is NOT a source of divergence. B09 (Programs mismatch) is therefore downstream in the
  ViewModel/UI/flow, NOT data — reclassify B09 as RUNTIME/flow.
- **WORKOUT-EXECUTION hotspot:** B02, B04, B05, B06, B07 all live in the workout flow (B03 removed). The execution
  screen + its ViewModel is the single most divergent feature. Split further into DATA (B03), PROGRESSION
  (B04/B06), WIRING (B07), VERSION (B05) — so it is not one bug, but the flow concentrates several.
- **WIRING / dead handlers:** B07 and B08 are the same failure shape — the call is translated but the
  behavior is lost. Worth checking whether they share a mechanism (e.g. a whole category of callback that
  doesn't fire in py).
- **PROGRESSION state machine:** B04 + B06 (advance-too-early / never-conclude) are two faces of one
  faulty set/exercise progression condition.
- **LAYOUT + RUNTIME:** B10 (symptom) is enabled by B11 (silent stub degradation). Fixing B11 (loud
  degradation) would turn future B10-class bugs from visual archaeology into named errors.

## Notes
- Tree-walking is PAUSED (owner call): it maps symptoms, not causes. Resume once causes are being fixed.
- Every fix must be a GENERAL transpiler/runtime fix (standing law: no hand-edits to transpiled py).
  A bug that can ONLY be fixed by hand-editing py is itself a finding about the transpiler's limits.
