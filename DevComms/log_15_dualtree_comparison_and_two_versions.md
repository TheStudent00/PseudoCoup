## Bottom line

1. A static **tree-comparison tool** now exists (`WFL_PseudoCoup/tools/dualgraph/`). It builds the
   UI object tree of BOTH apps from source — Kotlin via the ingest parser, PseudoCoup via Python
   `ast` — aligns them, and lists every object that has no equivalent on the other side. No
   emulator, no render, no recording. Runs all 30 screens into `build/dualgraph/REPORT.md`.
2. The transpiler **works**. On a real screen it faithfully maps the view STRUCTURE to the
   PseudoCoup widget vocabulary, and FLAGs everything it can't resolve (runtime data, nav, handler
   bodies) instead of guessing. Shown below, live output.
3. The thing causing the confusion: **there are two PseudoCoup versions of every screen.** The
   transpiler-generated one (`build/ingested/`, faithful but flag-riddled) and the hand-written one
   (`src/ui/`, which actually ships). The hand screens were written first; the transpiler was built
   afterward as a mirror/checker; **we never regenerated the screens from it.** So every divergence
   the tool reports is a hand-vs-blueprint *interpretation* gap, not a transpiler bug.
4. This raises a process decision that's yours (section "The decision").

Read that: `Text → define_text`, `OutlinedTextField → define_input_zone`, `Spacer →
define_spacer_zone`, `Button → button`, and the custom composables map through the kit —
`ResultMessage → result_message`, `SingleChoiceSegmentedButtonRow → segmented_row`,
`LabeledField → labeled_field`, `TopAppBar → top_bar`. The names it chose are PseudoCoup's
vocabulary, and they match what the hand screen imports. **It transpiled.**

What it could NOT do is everything behind the ViewModel: the `body` strings built by `"..." + "..."`
concatenation (FLAGged non-literal), `BugSeverity.entries` (FLAGged runtime list — it can't unroll a
list it can't see), and the `onClick` destinations / handler bodies (FLAGged, empty, no guess).

## The contrast, in one place

```
element        transpiler-generated                hand-written (ships)             why they differ
-----------    --------------------------------    -----------------------------    -----------------------------
intro text     define_text(..., "body_muted")      note(...)                        hand used a widget, not a primitive
severity ctrl  segmented_row(..., [], 0)  FLAGGED  define_box "seg_row" + while-     hand HAD the runtime data and
                                                   loop of define_button per item    unrolled it; also inlined to keep
                                                                                      its handler-id wiring (its NOTE)
handlers       return  # FLAG no guess             real on_back / on_severity ...    hand knows the nav + logic
```

So the divergence is two things stacked: (1) the transpiler's inherent FLAGs — the data/logic layer
it structurally can't see — which someone has to fill; and (2) the hand-author's interpretation
choices WHILE filling them (`note` vs `define_text`; inlined control vs `segmented_row`).

## "PC-only" almost never means PseudoCoup invented something

You asked: how can an object exist in PseudoCoup but not in Kotlin? Three buckets, by frequency:

1. **Exists in Kotlin, just unbridged.** `EquipmentRow` (Kotlin) ⇄ `list_item` (PseudoCoup) — same
   object. The ingest map (`MappingTable.KNOWN_CUSTOM`) has `ListItem → list_item` but not
   `EquipmentRow → list_item`, so the tool keeps `EquipmentRow` as one coarse node and can't link it
   to `list_item`; both show as one-halves. This is the only bucket where something genuinely DIDN'T
   transpile — the map is incomplete. Fix = add the bridge entry.
2. **My tool dropping it.** PseudoCoup renders the top bar as a content node (`top_bar`), but the
   Kotlin extractor deliberately SKIPS the Scaffold's `topBar` chrome, so there is no Kotlin node to
   match and `top_bar` reads PC-only on nearly every screen. Tool gap, not a real object. Fix = emit
   the `TopAppBar` as a node.
3. **Genuinely PseudoCoup-only (small, real).** Defensive scaffolding the hand-author added — e.g.
   `empty_state 'Gym not found'` — where Compose handles the null case via navigation, so no literal
   blueprint object exists.

One wrinkle worth knowing before any bridging: it is not always a clean alias. The blueprint's
`FilterChip` IS mapped (`FilterChip → equip_chip`), but the hand screens use a `chip` widget there,
not `equip_chip` — the transpiler-author and the hand-author chose DIFFERENT PseudoCoup widgets for
the same Kotlin concept. Bridging that means reconciling a real vocabulary inconsistency, not just
adding an alias.

## The decision (yours)

The transpiler maps STRUCTURE faithfully but cannot ship alone (the FLAGs are the data/nav/logic).
The hand screens have that logic but carry interpretation divergences. So:

- **(a) Keep the hand screens, reconcile divergences via the dualgraph tool, one at a time.** What
  we've been doing. Lowest churn; the interpretation gaps get closed by hand.
- **(b) Regenerate screens from the transpiler, then re-fill the FLAGs.** Closes the structural
  interpretation gaps by construction, but re-does the logic wiring the hand screens already have,
  and depends on the map being complete (it isn't yet — bucket 1 above).
- **(c) Hybrid — regenerate the screens that diverged badly, keep the faithful ones.**
